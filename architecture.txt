2. GIẢI THÍCH PHẦN ROOT: Library-system
==================================================

conote-system là thư mục gốc của toàn bộ project.

Nó không phải nơi để viết code nghiệp vụ chính.
Nó đóng vai trò là “project cha” để quản lý các module con.

Trong thư mục này thường chỉ nên có:

file pom.xml gốc
các thư mục module con
file cấu hình chung nếu cần

Không nên để:

src/main/java ở root
code Java nghiệp vụ ở root

Lý do:
Vì project đang làm theo Maven multi-module.
Root chỉ làm nhiệm vụ quản lý, còn code thật sẽ nằm trong từng module riêng.

==================================================
3. GIẢI THÍCH FILE ROOT pom.xml
==================================================

File:
conote-system/pom.xml

Vai trò:

là pom cha của toàn bộ hệ thống
quản lý version chung
quản lý dependency version chung
khai báo các module con
quy định Java version toàn project

Ví dụ root pom sẽ khai báo:

conote-common
conote-server
conote-client

Nghĩa là khi chạy:
mvn clean install

Maven sẽ build lần lượt:

conote-common
conote-server
conote-client

Tóm lại:
pom.xml ở root giống như “bộ chỉ huy trung tâm”.

5. GIẢI THÍCH MODULE library-common
==================================================
Thư mục:
library-common/

Vai trò:
Đây là module chứa những thành phần dùng chung cho cả client và server.

Nó chứa:

model
dto
enum

Mục đích:
Khi client và server đều cần dùng chung cấu trúc dữ liệu thì chỉ cần đặt trong common.

Ví dụ:

User
Book
Comment
Message
LoginRequest
LoginResponse
RegisterRequest
ApiResponse
Role
LanguageCode

Nhờ vậy:

không phải tạo User bên client và User bên server riêng
dữ liệu đồng bộ
dễ convert sang JSON
dễ bảo trì

7. GIẢI THÍCH PACKAGE TRONG library-common

7.1. dto/

DTO = Data Transfer Object

Nơi chứa các đối tượng dùng để truyền dữ liệu giữa client và server.

Ví dụ:

LoginRequest
LoginResponse
RegisterRequest
BookUploadRequest
ChatMessageRequest
ApiResponse<T>

Tại sao cần DTO:
Vì dữ liệu gửi qua mạng không nên gửi lung tung.
Cần có format rõ ràng.

Ví dụ:
LoginRequest có thể gồm:

username
password

LoginResponse có thể gồm:

success
message
userId
role

DTO rất quan trọng trong mô hình client-server.

7.2. model/

Chứa các model nghiệp vụ chính của hệ thống.

Ví dụ:

User
Book
Comment
Favorite
Message
Vocabulary
Translation
UserSetting

Model biểu diễn thực thể trong hệ thống.

Ví dụ:
User đại diện cho người dùng
Book đại diện cho sách
Comment đại diện cho bình luận

Lưu ý:
Model khác DTO ở chỗ:

Model thể hiện thực thể nghiệp vụ
DTO thể hiện dữ liệu truyền giữa các lớp hoặc qua mạng

7.3. enums/

Chứa các kiểu liệt kê.

Ví dụ:

Role
LanguageCode
MessageType
BookCategory
ThemeMode

Ví dụ:
Role có thể gồm:

USER
ADMIN

LanguageCode có thể gồm:

VI
EN
JA
KO

Dùng enum giúp code rõ ràng hơn, tránh dùng String bừa bãi.

7.4. util/

Chứa các tiện ích dùng chung.

Ví dụ:

JsonUtil
DateTimeUtil
StringUtil
ValidationUtil

Lưu ý:
Chỉ nên để các hàm tiện ích nhỏ, dùng chung.
Không nên nhét logic nghiệp vụ nặng vào util.

10. GIẢI THÍCH PACKAGE TRONG library-server

10.1. config/

Chứa các class cấu hình hệ thống.

Ví dụ:

DatabaseConfig
ServerConfig
SocketConfig

Mục đích:
Tập trung phần cấu hình vào một nơi để dễ quản lý.

Ví dụ:
DatabaseConfig có thể đọc thông tin:

database url
username
password

10.2. controller/

Chứa lớp tiếp nhận request và điều phối xử lý.

Trong mô hình của bạn, controller phía server là nơi nhận dữ liệu từ client.

Ví dụ:

AuthController
BookController
CommentController
FavoriteController
ChatController
TranslationController

Nhiệm vụ của controller:

nhận request
kiểm tra sơ bộ
gọi service tương ứng
trả response

Controller không nên chứa toàn bộ business logic.

Ví dụ:
Khi client gửi yêu cầu đăng nhập:

AuthController nhận LoginRequest
gọi AuthService xử lý
trả LoginResponse

10.3. service/

Chứa logic nghiệp vụ chính.

Đây là lớp rất quan trọng trong N-layer.

Ví dụ:

AuthService
BookService
CommentService
FavoriteService
VocabularyService
TranslationService
ChatService

Nhiệm vụ:

xử lý luật nghiệp vụ
kết hợp nhiều DAO
kiểm tra logic hệ thống

Ví dụ:
BookService có thể:

lưu thông tin sách
gọi bộ đọc PDF/Word
lưu text đã trích xuất
kiểm tra sách hợp lệ

Tóm lại:
service là nơi quyết định hệ thống hoạt động như thế nào.

10.4. dao/

DAO = Data Access Object

Đây là lớp truy cập dữ liệu từ database.

Ví dụ:

UserDAO
BookDAO
CommentDAO
FavoriteDAO
MessageDAO
VocabularyDAO

Nhiệm vụ:

viết SQL
insert/update/delete/select dữ liệu

Ví dụ:
UserDAO có thể có:

findByUsername()
save()
updatePassword()
findById()

DAO không nên xử lý nghiệp vụ phức tạp.
Nó chỉ nên tập trung vào database.

10.5. network/

Chứa phần giao tiếp mạng giữa client và server.

Ví dụ:

SocketServer
ClientHandler
RequestDispatcher
SessionManager

Nhiệm vụ:

mở cổng server
chấp nhận kết nối client
nhận dữ liệu JSON
chuyển request tới controller
gửi response ngược lại

Nếu bạn làm nhắn tin realtime, phần này rất quan trọng.

Ví dụ:
Client A gửi tin nhắn
-> SocketServer nhận
-> ChatController/ChatService xử lý
-> gửi tới Client B

10.6. security/

Chứa phần bảo mật.

Ví dụ:

BCryptUtil
PasswordEncoder
AuthSessionManager
TokenUtil nếu sau này mở rộng

Nhiệm vụ:

hash mật khẩu
so sánh mật khẩu khi đăng nhập
quản lý session đăng nhập

Ví dụ:
Khi đăng ký:

password gốc không lưu trực tiếp
BCrypt hash password
lưu hash vào DB

Khi đăng nhập:

lấy hash trong DB
so khớp bằng BCrypt

10.7. file/

Chứa xử lý file sách.

Đây là package rất quan trọng với đề tài của bạn.

Ví dụ:

FileStorageService
PdfExtractor
WordExtractor
FileValidator

Nhiệm vụ:

lưu file upload
kiểm tra loại file
trích xuất text từ PDF/DOCX
kiểm tra kích thước file

Ví dụ:
User upload sách PDF
-> FileStorageService lưu file
-> PdfExtractor đọc nội dung
-> BookService lưu text vào DB

10.8. util/

Chứa các hàm tiện ích riêng cho server.

Ví dụ:

FileNameUtil
JsonResponseUtil
ServerTimeUtil

Không nên để logic nghiệp vụ chính ở đây.

10.9. ServerApplication.java

Đây là class chạy chính của server.

Nhiệm vụ:

khởi động server
mở socket
nạp cấu hình
in log server started

Ví dụ:
public static void main(String[] args)

Đây là điểm bắt đầu của server.

18. LIÊN HỆ VỚI KIẾN TRÚC N-LAYER

Trong project của bạn, N-layer chủ yếu thể hiện rõ ở server:

Presentation Layer
controller
Business Layer
service
Data Access Layer
dao
Database Layer
MySQL / MariaDB

Phía client có thể hiểu đơn giản là:

UI Layer: FXML + controller
Client Service Layer: service
Communication Layer: network

Như vậy cấu trúc thư mục phản ánh đúng kiến trúc hệ thống.
